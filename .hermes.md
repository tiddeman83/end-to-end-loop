# Hermes Project Context

This repository is ready for Hermes Agent maintenance. It contains an Agent Skills
compatible `SKILL.md`, validation script, CI workflow, research paper draft, and
DevBoss handoff instructions.

## Hermes Operating Rules

- Load `SKILL.md` before making coding or release changes.
- Enable `skills.write_approval: true` before allowing Hermes to update installed
  skill copies.
- Use `AGENTS.md` as the general project bridge and `handoff/` for DevBoss office
  setup.
- Treat the user as Supervisory Board chair. New release plans require board
  approval before implementation.
- Route board decisions through the user's AI Assistant into Todoist, mirror them
  over Telegram when the origin supports it, then mirror them back into the DevBoss
  decision log.
- Poll Todoist for tasks prefixed `DevBoss ::`; treat them as user-supplied
  DevBoss office instructions/questions.
- Do not perform Firebase deploys until the user provides the environment and
  explicit deploy approval.

## Self-Learning Memory

- Before repo work, inspect `.end-to-end-loop/memory.md` and
  `.end-to-end-loop/memory.local.md` if present.
- After material repo work, write a compact result log and update memory when file
  writes are in scope.
- Keep memory CAVEMAN ULTRA compact: `FACT`, `CMD`, `BLOCK`, `PREF`, `RISK`,
  `FIX`, `AVOID`, `NEXT`.
- Never store secrets, tokens, private chat text, raw env values, or bulky logs.
- If uncertain whether a learning is safe to commit, write it to local memory or
  ask the user.
- When pushing to GitHub, include Copilot findings if the local environment exposes
  an authenticated Copilot/gh-copilot review path; otherwise record why it was not
  available.
